## 0. 更新版方案概述：双视角 AI + 茶文旅服务智能体

本项目由单一茶园采摘规划 agent 调整为“双视角”Web agent。用户进入系统后先选择使用视角：

- 游客视角：面向来访游客，提供羊楼洞历史文化、青砖茶与万里茶道介绍、视频和外部资料链接、从出发到住宿的行程规划，以及茶园采摘体验规划。
- 茶厂视角：面向茶厂、合作社和茶园管理者，利用 NDVI、无人机影像、冠层颜色和人工记录判断茶叶成熟程度，并规划按地块号执行的采摘路径。

第一版系统采用本地演示数据和外部链接，不直接接入 12306、携程或 Trip.com 的实时订票、订房、支付和库存接口。12306、携程、Trip.com、视频资料等以可点击外链方式接入，用于演示完整服务流程。

### 0.1 游客视角功能

游客视角重点解决“来之前了解什么、怎么来、住哪里、怎么玩、能否体验采摘”的问题。功能包括：

- 羊楼洞历史文化介绍。
- 青砖茶、万里茶道、古街和茶产业知识卡片。
- 羊楼洞相关视频、图文和搜索资料外链。
- 12306 火车票查询入口。
- 携程、Trip.com 等住宿和行程查询入口。
- “古街—中国青砖茶博物馆—万亩茶园体验区—茶文创产品店”的行程建议。
- 茶园采摘体验规划，包括体验时间、体验地块、路线和注意事项。

### 0.2 茶厂视角功能

茶厂视角保留并强化原有茶园生产规划功能，重点解决“哪些地块成熟、为什么成熟、先采哪里、怎么走、哪些数据还缺”的问题。功能包括：

- 以地块号为核心的茶园管理。
- 基于 NDVI、无人机 RGB/多光谱影像、冠层颜色和人工记录的成熟度判断。
- 展示成熟度等级、NDVI、无人机观测、判断置信度和数据缺口。
- 生成 P1-P4 采摘优先级。
- 生成采茶路径：起点 → 地块号顺序 → 加工/运输点。
- 标记缺少最新 NDVI 或无人机复查的地块，例如 YLD-C009 不直接进入采摘路径。
- 输出无人机复查任务和订单供货建议。

## 1. 项目目标

1. 以“地块号”为核心管理茶园，不使用模糊地名描述茶园位置。
2. 利用无人机影像、遥感指标和人工记录判断不同地块的茶叶成熟程度。
3. 根据成熟度、预计产量、采摘窗口、人力安排生成采摘优先级。
4. 输出未来 1 至 3 天的采摘顺序，帮助采摘队伍按地块执行任务。
5. 判断已经售出或预售的茶叶订单可以由哪些地块供货。
6. 规划无人机复查任务，优先检查成熟度不确定、数据过期或订单紧急的地块。
7. 形成可演示的 agent 原型，支持自然语言问答和规划结果解释。

## 2. Agent 定位

Agent 需要回答的问题包括：

- 哪些地块已经成熟，可以进入采摘计划？
- 哪些地块还未成熟，需要继续观察？
- 明天如果采摘人手有限，应先采哪些地块？
- 某个商家已经卖出一批茶叶，应从哪些地块优先供货？
- 如果某个地块产量不足，是否有替代地块？
- 万亩茶园中哪些地块需要无人机优先复查？
- 某个地块为什么被排在采摘顺序第一位？

回答必须以地块号为核心，并给出判断依据。

## 3. 核心用户

### 3.1 茶园管理者

查看各地块成熟度、安排采摘顺序、调配采摘人员、协调运输和加工。

### 3.2 采摘队伍

根据 agent 输出的地块号、采摘顺序和任务清单执行采摘。

### 3.3 茶叶商家

根据可供应地块、预计产量、成熟度和采摘时间判断是否可以售卖、预售或接单。

### 3.4 合作社或茶企

统筹多片茶园、多批订单、多支采摘队伍和无人机巡检任务。

## 4. 所需设施与条件

### 4.1 茶园地块编号体系

需要先建立统一地块号体系。每个茶园地块必须有唯一编号，例如：

- YLD-A001
- YLD-A002
- YLD-B015
- YLD-C028

每个地块号需要对应固定边界、面积、品种、管理责任人和历史采摘记录。所有 agent 输出都应优先使用地块号，不使用“山上那片”“东边茶园”等无法执行的描述。

### 4.2 无人机巡检设备

需要用于茶园巡检的无人机设备，支持对万亩茶园分区航拍。基础要求包括：

- 可采集 RGB 影像。
- 可按地块号或航线任务执行巡检。
- 可记录拍摄时间、航线、地块范围和影像文件。
- 条件允许时配备多光谱相机，用于 NDVI、GNDVI 等植被指数分析。

### 4.3 遥感与影像处理条件

需要具备基础影像处理能力，用于从无人机影像中识别茶园状态。可包括：

- 地块边界标注工具。
- 正射影像或航拍图拼接工具。
- 植被指数计算工具。
- 茶树冠层颜色、纹理、覆盖度分析工具。
- 成熟度人工标注样本，用于校正识别结果。

### 4.4 数据采集终端

需要手机、平板或电脑用于现场记录和数据录入。采集内容包括：

- 地块号
- 巡检日期
- 人工成熟度判断
- 预计产量
- 采摘人员数量
- 天气情况
- 商家订单信息
- 现场备注和照片

### 4.5 订单与销售记录

需要商家或茶企提供订单数据，用于判断哪些茶源已经售出或需要优先采摘。订单数据至少包括：

- 商家名称或商家编号
- 订单数量
- 茶叶类型或品质要求
- 最晚交付时间
- 是否指定地块或等级
- 当前订单状态
- 可替代供货要求

### 4.6 Agent 原型运行环境

需要一个轻量化系统环境支持 agent 演示和测试，可采用网页、表格加聊天界面、小程序原型或本地演示工具。基础条件包括：

- 可导入地块数据表。
- 可导入无人机巡检结果。
- 可导入订单数据。
- 可输出采摘顺序表和无人机复查表。
- 可通过自然语言提问并获得解释性回答。

## 5. 核心数据要求

### 5.1 茶园地块基础信息

| 字段 | 说明 |
|---|---|
| 地块号 | 唯一编号，作为所有规划的核心索引 |
| 地块边界 | 可由地图或无人机航测生成 |
| 面积 | 用于估算产量和采摘工作量 |
| 茶树品种 | 影响成熟时间和采摘标准 |
| 历史产量 | 用于估算本次可采摘量 |
| 最近巡检时间 | 判断数据是否需要更新 |
| 当前成熟度 | 由遥感识别、人工记录或综合模型判断 |
| 预计可采摘日期 | 用于生成采摘窗口 |
| 订单绑定状态 | 判断该地块是否已被商家预定或部分售出 |

### 5.2 无人机与遥感数据

无人机需对万亩茶园进行巡检，采集不同地块的影像数据，用于判断茶叶生长状态和成熟程度。

无人机采集数据包括：

- RGB 影像
- 多光谱影像，如条件允许可采集 NDVI、GNDVI 等植被指数
- 地块边界影像
- 茶园冠层颜色、纹理和覆盖度
- 异常区域，如长势弱、病虫害疑似区、采摘遗漏区

### 5.3 商家销售与订单数据

商家可能已经将部分茶叶售出或预售，因此采摘规划不能只看成熟度，还需要结合订单需求。

订单数据包括：

- 商家名称或商家编号
- 订单数量
- 需要交付的茶叶类型
- 最晚交付时间
- 是否指定地块或品质等级
- 当前订单状态
- 可替代供货地块

## 6. Agent 功能要求

### 6.1 地块成熟度识别

Agent 需要根据无人机影像、遥感指标和人工记录判断每个地块的成熟程度。成熟程度建议分为：

- 未成熟
- 接近成熟
- 已成熟
- 过熟风险
- 需复查

输出时应包含成熟度等级、判断依据和建议动作。例如：

> 地块 YLD-A012 当前为“已成熟”，冠层颜色和长势指标达到采摘标准，建议 24 至 48 小时内安排采摘。

### 6.2 采摘优先级排序

Agent 需要综合以下因素生成采摘优先级：

- 成熟程度
- 是否存在过熟风险
- 地块面积
- 预计产量
- 采摘所需人力
- 到加工点或运输点的距离
- 商家订单交付时间
- 天气和无人机复查结果

优先级建议分为：

- P1：必须优先采摘
- P2：建议近期采摘
- P3：继续观察
- P4：暂不采摘

### 6.3 采摘顺序规划

Agent 需要输出明确的采摘顺序，而不是只给出成熟度列表。

示例输出：

| 顺序 | 地块号 | 成熟度 | 预计产量 | 建议时间 | 原因 |
|---|---|---|---|---|---|
| 1 | YLD-A012 | 已成熟 | 约 280 kg | 明日上午 | 已成熟且绑定订单 |
| 2 | YLD-A018 | 过熟风险 | 约 190 kg | 明日下午 | 成熟窗口较短 |
| 3 | YLD-B006 | 接近成熟 | 约 320 kg | 后天上午 | 可满足后续订单 |

### 6.4 订单与茶源匹配

当商家已经卖出或预售茶叶时，Agent 需要判断哪些地块可以供货，并给出匹配建议。

示例问题：

- 商家 A 已经卖出 500 kg，哪些地块可以满足？
- 如果 YLD-A012 产量不足，能否用其他地块补充？
- 哪些地块适合做高品质订单？

Agent 输出应包括：

- 推荐供货地块
- 预计可供数量
- 采摘时间
- 订单满足程度
- 替代地块方案

### 6.5 无人机巡检任务规划

Agent 需要根据成熟度不确定、数据过期或订单紧急程度，安排无人机复查任务。

示例输出：

| 巡检优先级 | 地块号 | 原因 | 建议动作 |
|---|---|---|---|
| 高 | YLD-C009 | 上次巡检已超过 3 天，且接近成熟 | 明早无人机复查 |
| 中 | YLD-B014 | 影像疑似长势不均 | 补拍多光谱影像 |
| 低 | YLD-A003 | 当前未成熟 | 3 天后复查 |

### 6.6 自然语言问答

Agent 需要支持茶园管理者和商家用自然语言提问，例如：

- 今天最应该采哪个地块？
- 明天如果下雨，采摘顺序要不要调整？
- 哪些地块可以供应已经卖出去的订单？
- YLD-A018 为什么排在第一？
- 无人机下一次应该飞哪些地块？

回答必须给出依据，不能只给结论。

## 7. 工作流程设计

### 7.1 每日采摘规划流程

1. 导入最新无人机巡检数据和人工记录。
2. 更新每个地块的成熟度等级。
3. 导入商家订单和预计交付时间。
4. 计算各地块采摘优先级。
5. 输出当天或未来三天的采摘顺序。
6. 标记需要无人机复查的地块。
7. 生成给管理者、采摘队伍和商家的不同版本任务清单。

### 7.2 商家售卖辅助流程

1. 商家输入已售出或计划售卖的数量、品质和交付时间。
2. Agent 查询成熟地块和预计产量。
3. Agent 推荐可供货地块。
4. 如果茶源不足，Agent 给出替代地块或延迟交付建议。
5. 商家确认后，系统将订单绑定到对应地块。

### 7.3 无人机复查流程

1. Agent 找出成熟度不确定或数据过期的地块。
2. 根据订单紧急程度和地块风险排序。
3. 输出无人机飞行巡检清单。
4. 巡检完成后更新成熟度判断。
5. 自动调整采摘顺序。

## 8. 原型系统模块

| 模块 | 功能 |
|---|---|
| 地块管理模块 | 录入和查看地块号、面积、边界、品种和历史数据 |
| 无人机数据模块 | 上传或接入巡检影像，更新成熟度判断 |
| 成熟度识别模块 | 对地块进行未成熟、接近成熟、已成熟、过熟风险等分类 |
| 采摘规划模块 | 自动生成采摘优先级和采摘顺序 |
| 订单匹配模块 | 将商家订单与可采摘地块进行匹配 |
| 巡检任务模块 | 规划无人机复查地块 |
| Agent 问答模块 | 用自然语言解释规划结果和回答管理问题 |
| 报表输出模块 | 输出每日采摘计划、供货建议和复查清单 |

## 10. 示例 Agent 要求清单

### 10.1 必须支持的输入

- 地块号
- 地块面积
- 茶树品种
- 无人机巡检日期
- 遥感成熟度指标
- 人工成熟度记录
- 预计产量
- 商家订单数量
- 订单交付时间
- 采摘人员数量
- 天气或现场备注

### 10.2 必须支持的输出

- 每个地块的成熟度等级
- 每个地块的采摘优先级
- 未来 1 至 3 天采摘顺序
- 商家订单可供货地块
- 茶源不足时的替代方案
- 无人机复查地块清单
- 建议依据和原因解释

### 10.3 回答原则

Agent 回答必须符合以下要求：

- 以地块号为核心，不使用模糊地名。
- 每个建议必须说明依据。
- 对不确定数据要提示“需复查”。
- 不夸大遥感识别准确性。
- 不直接替代人工决策，而是提供辅助规划建议。
- 对已经售出或绑定订单的茶源要明确标注。
